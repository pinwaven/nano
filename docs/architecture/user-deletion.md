# User Deletion Lifecycle

When a user is deleted from the Waven Nano platform via the Admin Panel, the system executes a direct `DELETE FROM users WHERE user_id = $1` query. The cleanup of associated data relies primarily on PostgreSQL's foreign key constraints rather than application-level logic.

This approach ensures absolute data consistency at the database level and eliminates race conditions or partial deletions.

## 1. Data Destroyed (`ON DELETE CASCADE`)

When the user record is deleted, all personal and health data owned exclusively by that user is permanently destroyed via `ON DELETE CASCADE`.

This includes:
* **Health & Assessment Data:** 
  - `biomarkers` (raw scan results and BioAge calculations)
  - `scans` (records linking a physical chip to the user)
* **Nutrition & Dispensing:** 
  - `nutrition_plans` (AI-generated 7-day plans)
  - `nutrition_schedules` (specific morning/evening schedules)
  - `user_cartridges` (Dots inventory)
  - `reminders` (Dispensing schedules)
* **Engagement & App State:** 
  - `chat_messages` (chat history with the AI)
  - `notifications` (pending/sent app notifications)
  - `user_identities` (WeChat linkage and session data)
* **E-commerce:**
  - `orders` (store orders placed by the user)
* **Onboarding:**
  - `invitation_uses` (records of which invite code the user consumed to join)
* **Questionnaires:**
  - `questionnaire_assignments` (assignments given to the user)
  - `questionnaire_responses` (all answers within those assignments, via cascade from assignments)

## 2. Data Preserved (`ON DELETE SET NULL`)

If the user acted as an administrator or creator on the platform, their created assets are preserved to maintain ecosystem integrity, but the association to the deleted user is nullified using `ON DELETE SET NULL`.

* **Invitations (`invitations.created_by`):** 
  Invite codes generated by the user remain active and can still be used. The `created_by` field becomes `NULL`.
* **Coach Profiles (`coaches.user_id`):** 
  If the user held the `coach` role, their underlying Coach Profile record remains intact. This preserves the historical assignment and reporting for any users they were coaching. However, the `user_id` linkage (which ties the profile to their WeChat login) is set to `NULL`.
* **Commissions & Payouts (`commissions.approved_by`):** 
  If the user was an admin who approved channel or coach payouts, the `approved_by` audit trail is set to `NULL`.
* **Questionnaire assignments (`questionnaire_assignments.assigned_by`):**
  If the user was a coach or admin who assigned questionnaires to others, those assignments are preserved but `assigned_by` is set to `NULL`.

## 3. Frontend Safeguards (Role Blocking)

Because deleting a user with elevated privileges (like an admin or a coach) can leave orphaned configuration records (like a Coach Profile with no login identity), the Admin Panel implements a strict safety mechanism.

**The "Delete User" action is blocked if the user holds any role other than `user`.**

* If a user's `roles` array contains `coach`, `admin`, or `superadmin`, the delete button in the Admin Panel is disabled.
* The UI displays a warning: `"Cannot delete: user has the roles [roles]. Remove all extra roles first."`
* An admin must explicitly demote the user by editing their profile and removing the extra roles before the user can be deleted.
